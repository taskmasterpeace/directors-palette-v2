# Directors Palette Public API v2 â€” Design Spec

**Date:** 2026-03-19
**Status:** Draft

---

## Overview

A REST API that exposes all Directors Palette capabilities externally via API key authentication. Callers can generate images (all models, LoRAs, img2img), generate videos, create character sheets, execute recipes, expand wildcards, and run batch jobs â€” all with the same pts-based billing as the web app.

**Design philosophy:** Build v2 routes inside the existing Next.js app, wrapping the internal services that already work. Clean REST conventions so the API contract could survive extraction to a standalone service later if needed.

**Cost unit:** All costs in the API are expressed as integer **pts** (1 pt = $0.01). Internally, `MODEL_CONFIGS.costPerImage` is in dollars â€” the conversion is `pts = Math.round(costPerImage * 100)`.

## 1. Authentication

### API Key Format
- Keys use existing `dp_XXXXX` format generated by `apiKeyService`
- Passed as `Authorization: Bearer dp_xxx`
- Single key = full access at launch
- Keys are SHA-256 hashed in the database; raw key shown once on creation
- **Scope handling:** v2 routes ignore existing v1 scopes. All valid, active keys get full v2 access. Scoped access is a future consideration (see Section 8).

### Auth Middleware
A single `validateV2ApiKey()` function in `src/app/api/v2/_lib/middleware.ts`:

1. Extract key from `Authorization` header via `apiKeyService.extractKeyFromHeader()`
2. Call `apiKeyService.validateApiKey(rawKey)` â€” checks hash, expiry, active status
3. If valid â†’ look up user email from `auth.users` table using the returned `userId`
4. Return `{ userId, email }` context
5. If invalid/missing â†’ `401 Unauthorized`
6. Log usage via `apiKeyService.logUsage()`

> **Note:** `validateApiKey()` returns `{ apiKey, userId, scopes }` but not `email`. The middleware must fetch email separately via a Supabase `auth.users` lookup.

### Pts Billing
- Every generation endpoint checks user's pts balance before creating a job
- If insufficient â†’ `402 Payment Required` with cost and current balance
- Pts deducted upfront when job is created (same as web app)
- Admin keys can optionally bypass billing (checked via `admin_users` table)

### Rate Limiting
- 60 requests/minute per API key
- Returns `429 Too Many Requests` with `Retry-After` header
- **Implementation:** In-memory counter for self-hosted deployments. Note: on serverless (Vercel), each isolate has its own counter â€” rate limiting is approximate. For strict enforcement, use the `api_usage` table with a time-windowed count query instead.

## 2. Response Format

All endpoints return a consistent JSON envelope:

```json
// Success
{
  "success": true,
  "data": { ... }
}

// Error
{
  "success": false,
  "error": {
    "code": "INSUFFICIENT_PTS",
    "message": "Need 4 pts, you have 2"
  }
}
```

### Standard Error Codes
| Code | HTTP Status | Meaning |
|------|-------------|---------|
| `UNAUTHORIZED` | 401 | Missing or invalid API key |
| `INSUFFICIENT_PTS` | 402 | Not enough pts for this operation |
| `FORBIDDEN` | 403 | Key valid but action not allowed |
| `NOT_FOUND` | 404 | Resource doesn't exist |
| `VALIDATION_ERROR` | 422 | Bad input (details in message) |
| `RATE_LIMITED` | 429 | Too many requests |
| `INTERNAL_ERROR` | 500 | Server error |

## 3. Async Job System

Image and video generation are async. All generation endpoints return a job immediately; callers poll for completion or provide a webhook URL.

### Job Lifecycle
```
pending â†’ processing â†’ completed
                    â†’ failed
```

### Poll Single Job
**`GET /api/v2/jobs/{jobId}`**

```json
{
  "success": true,
  "data": {
    "job_id": "job_abc123",
    "status": "completed",
    "type": "image",
    "cost": 4,
    "created_at": "2026-03-19T10:00:00Z",
    "completed_at": "2026-03-19T10:00:12Z",
    "result": {
      "images": [
        { "url": "https://...", "width": 1024, "height": 1024 }
      ]
    }
  }
}
```

When `status` is `pending` or `processing`, `result` is `null`. When `failed`, an `error_message` field is included.

### List Jobs
**`GET /api/v2/jobs`**

Query params: `status` (filter), `type` (filter), `limit` (default 20, max 100), `offset` (default 0).

Returns the caller's recent jobs, newest first.

### Webhook (Optional)
Callers can include `"webhook_url": "https://..."` in any generation request. When the job completes or fails, a POST is sent to that URL with the same job payload. Webhook delivery is best-effort (single attempt, no retries at launch).

### Implementation
Jobs map to the existing gallery + prediction system:
- A new `api_jobs` table maps external job IDs to internal prediction/gallery IDs
- Status tracked by Replicate webhook â†’ gallery table update â†’ `api_jobs` update
- Poll endpoint reads from `api_jobs` table (joined with gallery for result URLs)
- **Webhook integration point:** The existing Replicate webhook handler at `src/app/api/webhooks/replicate/route.ts` is modified to also check `api_jobs` for matching `prediction_id`. If found and `webhook_url` is set, fire a POST with the job payload after the gallery update.

## 4. Endpoints

### 4.1 Image Generation

**`POST /api/v2/images/generate`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | string | yes | Model ID (see `/v2/models`) |
| `prompt` | string | yes | Generation prompt |
| `width` | number | no | Image width (default from aspect ratio) |
| `height` | number | no | Image height (default from aspect ratio) |
| `aspect_ratio` | string | no | e.g. "16:9", "1:1" (default "1:1") |
| `loras` | array | no | `[{ "id": "nava", "strength": 0.8 }]` |
| `reference_image` | string | no | URL â€” triggers img2img mode |
| `reference_strength` | number | no | 0.0-1.0 (default 0.75) |
| `num_images` | number | no | 1-5 (default 1) |
| `seed` | number | no | Reproducibility seed |
| `webhook_url` | string | no | URL for completion callback |

> **Note on `num_images`:** The API caps at 5 per request to keep response times reasonable. The internal config allows up to 15, but batch endpoint is the better path for bulk generation.

**Response:**
```json
{
  "success": true,
  "data": {
    "job_id": "job_abc123",
    "status": "pending",
    "cost": 4,
    "poll_url": "/api/v2/jobs/job_abc123"
  }
}
```

**Available image models:**
| Model ID | Type | Cost (pts) | LoRA Support | img2img | Max Refs |
|----------|------|------------|--------------|---------|----------|
| `nano-banana-2` | generation | 6 | no | yes | 14 |
| `z-image-turbo` | generation | 3 | no | yes | 1 |
| `flux-2-klein-9b` | generation | 4 | yes | yes (via fal.ai edit endpoint) | 5 |
| `firered-image-edit` | editing | varies | no | yes (required) | 1 |
| `qwen-image-edit` | editing | varies | no | yes (required) | 1 |

> **Editing models** (`firered-image-edit`, `qwen-image-edit`) require `reference_image` â€” they edit an existing image rather than generating from scratch. The `prompt` describes the desired edit.

**Internal wiring:** Calls `imageGenerationService.generateImage()` with the same parameters the web UI uses. LoRA validation is per-model (Klein 9B supports LoRAs). Reference image triggers the appropriate endpoint variant (e.g., fal.ai edit endpoint for Klein 9B).

### 4.2 Video Generation

**`POST /api/v2/videos/generate`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | string | no | Animation model (default "wan-2.2-5b-fast") |
| `prompt` | string | yes | Motion/camera prompt |
| `source_image` | string | yes* | Source image URL (*optional for seedance-1.5-pro) |
| `duration` | number | no | Duration in seconds (default 5) |
| `aspect_ratio` | string | no | e.g. "16:9" |
| `fps` | number | no | Frames per second (model default) |
| `resolution` | string | no | e.g. "720p" (model default) |
| `camera_fixed` | boolean | no | Lock camera position (default false) |
| `webhook_url` | string | no | Completion callback |

**Available video models:**
| Model ID | Notes |
|----------|-------|
| `wan-2.2-5b-fast` | Default, fast img2vid |
| `wan-2.2-i2v-fast` | Image-to-video variant |
| `seedance-lite` | Lightweight, supports 1-4 reference images |
| `seedance-pro-fast` | High quality, fast |
| `seedance-1.5-pro` | Supports text-to-video (no source image needed) |
| `p-video` | |
| `kling-2.5-turbo-pro` | High quality |

**Internal wiring:** The v2 route maps flat API parameters into the `ModelSettings` shape that `VideoGenerationService.buildReplicateInput()` expects: `{ duration, fps, resolution, cameraFixed }` etc. Cost calculated via `VideoGenerationService.calculateCost()`. Videos stored in Cloudflare R2.

### 4.3 Character Sheet Generation

**`POST /api/v2/characters/generate`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Character name |
| `description` | string | yes | Character appearance description |
| `style` | string | no | Style preset name |
| `reference_image` | string | no | Character reference URL |
| `style_reference` | string | no | Style reference URL |
| `aspect_ratio` | string | no | Default "16:9" |
| `webhook_url` | string | no | Completion callback |

**Response:** Returns **two jobs** â€” one for the turnaround sheet (front/side/back/three-quarter views) and one for the expression sheet (neutral/happy/sad/angry/surprised/thinking).

```json
{
  "success": true,
  "data": {
    "turnaround": {
      "job_id": "job_abc",
      "status": "pending",
      "cost": 6,
      "poll_url": "/api/v2/jobs/job_abc"
    },
    "expressions": {
      "job_id": "job_def",
      "status": "pending",
      "cost": 6,
      "poll_url": "/api/v2/jobs/job_def"
    },
    "total_cost": 12
  }
}
```

**Internal wiring:** Wraps `CharacterSheetService.generateCharacterSheet()` which always generates both sides (`side1` = turnaround, `side2` = expressions). The `description` is injected into the default templates as the character description. Uses Nano Banana Pro model. Two separate predictions are created, each tracked as its own job.

### 4.4 Recipe Execution

**`POST /api/v2/recipes/execute`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `recipe_id` | string | yes | Recipe ID from `/v2/recipes` |
| `fields` | object | yes | Key-value map filling template slots |
| `model` | string | no | Override model (default from recipe) |
| `aspect_ratio` | string | no | Override aspect ratio |
| `reference_images` | array | no | URLs for reference-based stages |
| `webhook_url` | string | no | Completion callback |

**Response:** Same async job format. Multi-stage recipes execute sequentially server-side (pipe chaining). The final result contains all stage outputs.

**Internal wiring:** Wraps `recipeExecutionService.executeRecipe()`. Validates all required fields from recipe template are provided. Supports generation, tool (remove-bg, grid-split), and analysis stages.

**`GET /api/v2/recipes?limit=50&offset=0`** â€” Lists the caller's available recipes with field definitions.

```json
{
```

### 4.5 Wildcard Expansion

**`POST /api/v2/wildcards/expand`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `prompt` | string | yes | Prompt with `_wildcard_name_` tokens |
| `count` | number | no | Number of expansions to return (default 1) |

**Response (synchronous â€” no job needed):**
```json
{
  "success": true,
  "data": {
    "expansions": [
      {
        "text": "a dwarf warrior wearing plate mail in a volcanic forge",
        "wildcards_used": ["fantasy_race", "armor_type", "setting"]
      }
    ]
  }
}
```

**Internal wiring:** Reads from user's wildcards in Supabase via `wildcardService`. Each `_name_` token is replaced with a random line from the matching wildcard's content. Multiple `count` returns different random picks.

**`GET /api/v2/wildcards?limit=50&offset=0`** â€” Lists the caller's wildcards with names, categories, and line counts.

### 4.6 Batch Generation

**`POST /api/v2/batch`**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `jobs` | array | yes | Array of job objects (max 20) |

Each job object is the same body you'd send to any generation endpoint, plus a `type` field. Supported types: `image`, `video`, `character`, `recipe`.

```json
{
  "jobs": [
    { "type": "image", "model": "nano-banana-2", "prompt": "..." },
    { "type": "image", "model": "z-image-turbo", "prompt": "..." },
    { "type": "video", "model": "wan-2.2-5b-fast", "prompt": "...", "source_image": "..." },
    { "type": "character", "name": "Elena", "description": "elven ranger" },
    { "type": "recipe", "recipe_id": "rec_abc", "fields": { "CHARACTER": "Elena" } }
  ]
}
```

**Response:**
```json
{
  "success": true,
  "data": {
    "batch_id": "batch_xyz",
    "total_cost": 36,
    "jobs": [
      { "job_id": "job_1", "status": "pending", "cost": 4 },
      { "job_id": "job_2", "status": "pending", "cost": 3 },
      { "job_id": "job_3", "status": "pending", "cost": 25 }
    ]
  }
}
```

Total cost is validated upfront â€” if the user can't afford the full batch, the entire request is rejected (no partial execution). Individual jobs are polled via `/v2/jobs/{id}` as usual.

## 5. Data Model

### New Table: `api_jobs`

Maps external API job IDs to internal prediction/gallery records.

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | External job ID (returned as `job_xxx`) |
| `user_id` | uuid (FK) | Owner |
| `api_key_id` | uuid (FK) | Key used to create the job |
| `type` | text | "image", "video", "character", "recipe" |
| `status` | text | "pending", "processing", "completed", "failed" |
| `prediction_id` | text | Internal Replicate prediction ID |
| `gallery_id` | uuid | Internal gallery record ID |
| `cost` | integer | Pts charged |
| `input` | jsonb | Original request body (for debugging) |
| `result` | jsonb | Output URLs, dimensions, etc. |
| `error_message` | text | Error details if failed |
| `webhook_url` | text | Caller's webhook URL (nullable) |
| `created_at` | timestamptz | |
| `completed_at` | timestamptz | |

**Indexes:**
- `prediction_id` â€” for webhook handler lookup (unique where not null)
- `(user_id, created_at DESC)` â€” for job listing/polling
- `(status)` â€” for filtering pending/processing jobs

**RLS:** Users can only read their own jobs (`user_id = auth.uid()`).

### Webhook Delivery

When a Replicate webhook fires and updates the gallery, the webhook handler at `src/app/api/webhooks/replicate/route.ts` adds an additional step: query `api_jobs` for a matching `prediction_id`. If found and `webhook_url` is set, fire a POST to that URL with the full job payload. This runs after the gallery update to ensure result URLs are populated.

## 7. Key Implementation Details

### Reference Image Handling
External URLs are passed directly to the generation services. The services already handle URL validation and upload to Replicate/Supabase as needed. Base64 data URIs are NOT supported in the API â€” callers must provide hosted URLs.

### LoRA Resolution
The `loras` array in image generation accepts either:
- Built-in LoRA ID (e.g., "nava-style") â€” resolved from the LoRA store
- Custom weights URL â€” passed directly to the model

### Video Parameter Mapping
The v2 route maps flat API parameters into the internal `ModelSettings` shape:
```
API: { duration, fps, resolution, camera_fixed }
  â†’ Internal: { duration, fps, resolution, cameraFixed }
```
Unspecified parameters use model-specific defaults from `VIDEO_MODEL_CONFIGS`.

### Recipe Field Validation
When executing a recipe, all fields marked `required: true` must be present. Select-type fields are validated against their options list. Wildcard-type fields are expanded before template substitution.

### Batch Cost Calculation
Batch endpoint sums the cost of all jobs, checks balance once. If affordable, all jobs are created and queued. Jobs execute independently (parallel where possible). Character jobs count as 2 sub-jobs (turnaround + expressions).

### Existing v1 Routes
v1 routes remain unchanged for backward compatibility. No migration needed â€” v2 is additive.
